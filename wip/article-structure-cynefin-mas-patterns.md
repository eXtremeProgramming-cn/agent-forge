## 第一部分:界定范畴 (Defining the Domain)

### 1.1 什么是"研究用MAS"?

**核心定义:**
- **系统边界**: 研究过程本身,而非研究对象
- **根本目的**: 知识发现与洞察生成
- **主要对象**: 非结构化质性材料(文献、档案、访谈、网络内容)
- **应用领域**: 社会科学、人文研究、区域国别研究、政策分析等

**与其他类型MAS的区别:**

| 维度 | 研究用MAS | 业务分析MAS(财务/市场) | 操作型MAS |
|------|----------|---------------------|---------|
| **目的** | 知识发现 | 决策支持 | 任务执行 |
| **输入** | 非结构化文本 | 结构化数据 | 明确指令 |
| **输出** | 洞察与理论 | 报告与建议 | 行动结果 |
| **评价标准** | 深度与原创性 | 准确性与及时性 | 效率与可靠性 |
| **人类角色** | 主导理论构建 | 审查与决策 | 监督与干预 |
| **不确定性** | 高(探索未知) | 中(预测趋势) | 低(执行已知) |

**边界澄清:**
- **包括**: 需要理解、综合、洞察生成的研究任务
- **不包括**: 纯数据驱动的实验研究(如生物医学实验)
- **灰色地带**: 定量社会科学研究(可能部分适用)

**关键特征:**
1. **信息的语义密度**: 需要深度理解语义,而非仅处理数据
2. **理论框架的中心性**: 研究质量依赖于合适的理论视角
3. **创造性知识生产**: 产生之前不存在的理论洞察
4. **迭代性认识论**: 理解随研究深入而演化

### 1.2 为什么需要一个新的设计框架?

**现有MAS设计方法的局限:**
- 技术架构关注"how to build"
- 缺乏对"what to build"的系统指导
- 没有考虑研究认识论的特殊性
- 忽视人机协作的边界问题

**Cynefin框架的价值:**
- 提供情境化的决策模式
- 区分不同类型的问题性质
- 指导合适的方法选择
- 明确人类智能介入的必要性

## 第二部分:概念框架 (Conceptual Framework)

### 2.1 引入"业务架构"概念

**为什么需要"业务架构"这个术语?**

之前的文章(declarative-multi-agent-architecture)讨论的是**技术架构**:
- Agent Blueprint的实现机制
- 三层分离(行为定义、知识、数据)
- 五大核心模式(Prompt-Defined Agent等)
- Runtime与Behavior的分离

但这些都没有回答:
- **一个研究任务应该分几个阶段?**
- **每个阶段做什么?顺序如何?**
- **哪些部分需要人类介入?**
- **数据如何在阶段间流动?**

这些问题属于另一个层次:**业务架构** (Research Process Architecture)

**业务架构的定义:**
> 研究任务的**逻辑分解**与**流程编排**,规定了完成研究目标所需的**阶段划分**、**任务序列**、**数据流向**、**人机协作点**,以及**知识转化路径**。

**业务架构 vs 技术架构:**

| 维度 | 业务架构 | 技术架构 |
|------|---------|---------|
| **关注点** | What & Why | How |
| **层次** | 研究逻辑 | 实现机制 |
| **例子** | "三阶段:采集→分析→报告" | "Prompt-Defined Agent模式" |
| **决定因素** | 研究问题性质(Cynefin域) | 工程原则与技术能力 |
| **设计者** | 研究者+AI架构师 | AI架构师 |
| **稳定性** | 因问题而异 | 相对稳定的模式 |

**关键洞察:**
- 技术架构可以**复用**(如AgentForge的声明式架构)
- 业务架构必须**定制**(根据研究问题的Cynefin域)
- 好的技术架构**使能**多种业务架构

### 2.2 Cynefin框架在研究情境中的重新诠释

**核心修正:重新定义"系统"和"行动"**

| Cynefin原始定义 | 研究情境下的重新诠释 |
|---------------|------------------|
| **系统** = 决策/管理环境 | **系统** = 研究过程本身 |
| **行动** = 组织干预/政策执行 | **行动** = 研究活动(尤其是数据创造) |

**为什么这个重新定义很重要?**
- 我们开发的研究用MAS**辅助研究过程**,通过研究**间接影响**研究对象
- 我们不是开发一个直接作用于研究对象的系统
- "系统"的边界是研究者的认知和知识生产过程

**四域的研究情境投射:**

#### Clear域:常规性信息处理
- **因果关系**: 明确且可预测
- **信息状态**: 结构化存在
- **研究活动**: 执行标准流程
- **决策模式**: Sense-Categorize-Respond
- **AI能力**: 全自动执行
- **人类角色**: 定义任务,验证结果
- **典型任务**: 下载文件、转录录音、格式转换

#### Complicated域:结构化分析
- **因果关系**: 存在但需要专家分析
- **信息状态**: 存在但需要解读
- **研究活动**: 应用分析框架
- **决策模式**: Sense-Analyze-Respond
- **AI能力**: 作为专家系统
- **人类角色**: 定义框架,审查质量
- **典型任务**: 指标评估、结构化编码、框架应用

#### Complex域:涌现性理论构建
- **因果关系**: 仅事后可见
- **信息状态**: 存在但需要整合理解
- **研究活动**: 探索识别模式
- **决策模式**: Probe-Sense-Respond
- **AI能力**: 辅助材料准备
- **人类角色**: 识别涌现模式,构建理论
- **典型任务**: 理论框架发现、编辑判断、综合洞察

#### Chaotic域:创造性数据生成
- **因果关系**: 不可辨识
- **信息状态**: 基本不存在
- **研究活动**: **创造新数据/新知识**
- **决策模式**: Act-Sense-Respond
- **AI能力**: 仅辅助Clear子任务
- **人类角色**: 主导数据创造(如田野调研)
- **典型任务**: 实地调研、原创性访谈、档案发掘

### 2.3 域分类的决策树

**如何判断一个研究任务属于哪个域?**

```
问题1: 完成任务所需的信息是否已经存在?
├─ 基本不存在 → Chaotic
│   └─ 需要创造性行动生成新数据
│       例:无先例可循的田野调研
│
└─ 已存在 → 继续判断
    │
    问题2: 因果关系/分析路径是否明确?
    ├─ 明确 → 继续判断
    │   │
    │   问题3: 是否需要专家级判断?
    │   ├─ 不需要 → Clear
    │   │   └─ 标准流程可处理
    │   │       例:下载文件、格式转换
    │   │
    │   └─ 需要 → Complicated
    │       └─ 需要分析但路径清晰
    │           例:应用预定义指标评估
    │
    └─ 不明确 → Complex
        └─ 需要探索,模式涌现
            例:构建理论框架、编辑判断
```

**关键区分标准总结:**

| 标准 | Clear | Complicated | Complex | Chaotic |
|------|-------|------------|---------|---------|
| **信息存在性** | ✓ | ✓ | ✓ | ✗ |
| **因果关系清晰** | ✓ | ✓ | ✗ | ✗ |
| **需要专家判断** | ✗ | ✓ | ✓ | N/A |
| **模式可预先确定** | ✓ | ✓ | ✗ | ✗ |
| **AI可主导** | ✓ | ✓ | ✗ | ✗ |

### 2.4 域降解(Domain Decomposition):研究的演化路径

**核心洞察:研究过程是一个域降解过程**

研究的自然演化路径通常是:**Chaotic → Complex → Complicated → Clear**

这个降解过程不是自动发生的,而是**通过人类智能介入和MAS系统设计**来实现的。

#### 2.4.1 降解的本质

**什么是域降解?**
- 将高不确定性的问题**逐步转化**为低不确定性的问题
- 每次降解都**增加了结构**,减少了不确定性
- 降解不是简化,而是**知识的积累和秩序的建立**

**为什么域降解对研究用MAS设计至关重要?**
1. **指导业务架构设计**: 不同域需要不同的MAS模式
2. **确定人机协作点**: 域转换通常需要人类介入
3. **优化研究效率**: 尽早降解到AI可主导的域
4. **保证研究质量**: 在合适的域应用合适的方法

#### 2.4.2 四个降解阶段

**阶段1: Chaotic → Complex (建立初步秩序)**

**触发条件**:
- 研究领域缺乏既有知识基础
- 关键信息不存在于文献/档案中
- 因果关系完全不清晰

**降解机制**:
- **人类行动**: 田野调研、深度访谈、档案发掘
- **创造新数据**: 生成之前不存在的一手材料
- **AI辅助角色**: 处理Clear子任务(转录、整理、初步编码)

**降解结果**:
- 积累了足够的一手数据
- 开始看到初步模式和关联
- 可以提出理论性问题
- 进入**Probe-Sense-Respond**模式

**MAS设计策略**:
- 人类绝对主导,AI最小化辅助
- 不要试图让AI主导数据创造
- 关注数据记录和管理的工具支持

**例子**: 榕江村超研究
- **Chaotic状态**: 不清楚数字技术如何助力乡村振兴,现有材料太粗浅
- **Act**: 实地调研,访谈基层干部和村民
- **Sense**: 在调研过程中识别关键变量(流量来源、转化机制等)
- **结果**: 进入Complex域,可以开始探索理论框架

---

**阶段2: Complex → Complicated (理论框架确立)**

**触发条件**:
- 有了足够的材料和数据
- 但分析路径不明确
- 需要理论视角才能"看见"模式

**降解机制**:
- **人类洞察**: 阅读材料、与专家对话、形成理论框架
- **识别涌现模式**: 看出关键矛盾、核心变量、因果机制
- **框架明确化**: 将洞察转化为可操作的分析框架

**降解结果**:
- 确定了适用的理论框架
- 分析路径变得清晰
- 可以定义具体的分析维度和指标
- 进入**Sense-Analyze-Respond**模式

**MAS设计策略**:
- **解耦人类洞察与AI执行**
- 人类完成框架构建后,任务转化为Complicated
- AI可以基于框架进行结构化分析

**域转换模式: "Complex-to-Complicated Transformation"**
```
[AI: Preliminary Research] → [Human: Framework Construction] → [AI: Structured Analysis]
      (Complex域)                  (洞察介入点)                    (Complicated域)
```

**例子**: 南非阶级分析
- **Complex状态**: 有左翼组织文章,但不知道用什么理论视角分析
- **Probe**: AI采集和整理conjunctural overview材料
- **Sense**: 人类阅读材料,与专家交流
- **Human Insight**: 确定"新型殖民主义与种族主义"理论框架
- **结果**: 进入Complicated域,AI可以基于这个框架进行深入分析

---

**阶段3: Complicated → Clear (标准化与自动化)**

**触发条件**:
- 分析框架已经成熟且稳定
- 分析步骤可以标准化
- 不再需要专家级判断(或判断点很少)

**降解机制**:
- **流程标准化**: 将专家判断转化为明确规则
- **工具化**: 开发可复用的分析工具/模板
- **最佳实践确立**: "好的做法"固化为"最佳实践"

**降解结果**:
- 分析变成执行标准流程
- 可以高度自动化
- 进入**Sense-Categorize-Respond**模式

**MAS设计策略**:
- 全自动化执行
- 人类角色缩减为质量抽查
- 关注效率和规模化

**注意**: 这一步不是总需要的
- 很多研究任务停留在Complicated域就足够了
- 过度标准化可能损失必要的灵活性
- 只有当任务高度重复时才值得降解到Clear

**例子**: DSI评估的潜在演化
- **当前**: Complicated域(需要判断如何评分)
- **潜在**: 如果评估了50+个国家,评分判断可能标准化
- **结果**: 部分指标可能降解到Clear域(简单规则即可)

---

#### 2.4.3 降解失败的常见模式

**过早降解** (Premature Simplification):
- **表现**: 在Complex域强行使用Complicated方法
- **例子**: 用通用理论框架分析所有国家(忽视特殊性)
- **后果**: 分析肤浅,抓不住要点
- **避免**: 耐心等待模式涌现,不要急于套用框架

**拒绝降解** (Complexity Fetish):
- **表现**: 可以降解到Complicated但坚持停留在Complex
- **例子**: 每次分析都重新"发现"框架,不固化知识
- **后果**: 效率低下,知识无法积累
- **避免**: 识别稳定模式后,及时标准化

**降解过度** (Over-Standardization):
- **表现**: 将Complicated降解到Clear,丧失必要判断力
- **例子**: 把所有评分变成机械规则,失去灵活性
- **后果**: 系统僵化,无法应对边界情况
- **避免**: 保留关键的专家判断点

**忽视Chaotic** (Skipping Fieldwork):
- **表现**: 在需要Chaotic域工作时,试图用Complex方法替代
- **例子**: 用二手材料分析,而实际需要田野调研
- **后果**: 研究建立在薄弱的信息基础上
- **避免**: 诚实评估信息的存在性和质量

## 第三部分:典型范例 (Exemplars as Patterns)

**模式语言的结构:**
- 每个exemplar包含:情境、问题、业务架构、MAS设计、具体例子、关键设计原则、变体

### 3.1 Clear域范例

#### Exemplar 1: 档案文件批量获取

**情境**: 需要从公开渠道获取大量结构化或半结构化文档

**问题**:
- 文件数量多,手动下载费时
- 需要标准化命名和组织
- 可能需要简单的格式转换

**业务架构**:
```
Single-Stage Pipeline:
[Download & Organize] → 输出:规整的文件集合
```

**MAS设计**:
- Agent数量:1个(或不需要MAS,单脚本即可)
- 并行度:无(或按文件并行)
- 人机协作点:无(全自动)
- 所在域:Clear

**具体例子**: 下载News on China所有历史期刊

**关键设计原则**:
1. 定义清晰的目标URL列表
2. 建立标准化命名规则
3. 明确存储位置和组织结构
4. 可以用Claude Code直接完成,不一定需要AgentForge

**变体**:
- 需要登录认证
- 需要处理反爬虫机制
- 需要增量更新(检查已下载)

**域降解关系**:
- 通常是从更高域降解下来的辅助任务
- 例:田野研究(Chaotic)中需要整理录音→转录(Clear)

---

### 3.2 Complicated域范例

#### Exemplar 2: 指标化评估系统

**情境**: 对一个或多个对象进行结构化、可比较的评估

**问题**:
- 有明确的评估维度和指标
- 每个指标需要收集特定数据
- 评分规则清晰但需要判断
- 需要生成结构化报告

**业务架构**:
```
Three-Stage Pipeline:
[Data Collection] → [Analysis & Scoring] → [Report Generation]
     ↓                      ↓                       ↓
   可并行                 可并行                   串行
  (按指标)               (按指标)                 (整合)
```

**MAS设计**:

**阶段1:数据采集**
- Agent类型:Research Agent(按指标组织)
- 并行策略:按指标并行(16个指标可并行采集)
- 输出:结构化JSON(按预定义schema)
- 注意:单个国家评估,但指标间互不干涉,可并行

**阶段2:分析评分**
- Agent类型:Analysis Agent(按指标)
- 输入:阶段1的JSON
- 规则:外部化的评分标准(references/)
- 输出:带分数的结构化数据
- 并行策略:指标间独立,可并行分析

**阶段3:报告生成**
- Agent类型:Report Writer
- 输入:阶段2的评分数据
- 标准:报告质量规范
- 输出:最终报告(Markdown/PDF)
- 串行执行:需要整合所有指标结果

**人机协作点**:
- **设计阶段**: 人类定义指标体系、评分规则
- **执行阶段**: AI全自动执行
- **审查阶段**: 人类验证结果合理性

**具体例子**: DSI (数字主权指数)评估
- 4维度,16指标
- 每指标15-20个细化问题
- 明确的评分规则
- 单个国家评估,访问数千个网页
- 按指标并行:16路并行采集和分析

**关键设计原则**:
1. **知识外部化**: 指标定义、评分规则放在references/
2. **数据契约**: 严格的JSON schema保证阶段间数据流
3. **质量规范嵌入**: Agent Blueprints包含质量标准
4. **支持并行**: 识别可并行的维度(如按指标),但不强制并行
5. **灵活执行**: 根据实际情况(网页数量、时间限制)决定并行度

**变体**:

**变体2a: 自适应问题生成**
```
Four-Stage Pipeline:
[Overview] → [Question Formulation] → [Data Collection] → [Analysis] → [Report]
```

**额外特征**:
- 第一阶段做粗粒度调研(Overview)
- 第二阶段根据对象特征生成定制化问题
- 例:尼泊尔→稻米问题,加纳→可可豆问题

**仍属Complicated因为**:
- 问题生成是规则引导的(基于Overview结果)
- 不需要人类洞察介入
- 过程是确定性的

**域降解关系**:
- 通常从Complex域降解而来
- 例:阶级分析中,理论框架确立后→指标化分析(Complicated)
- 也可能进一步降解到Clear(如果高度标准化)

### 3.3 Complex域范例

#### Exemplar 3a: 理论框架发现型研究

**情境**: 需要为特定研究对象构建适用的理论框架,通用框架不适用

**问题**:
- 研究对象的关键特征和矛盾点不明显
- 需要理论视角才能抓住重点
- 理论框架无法预先确定
- 框架从探索中涌现

**业务架构**:
```
Mixed Human-AI Pipeline (域转换模式):
[Preliminary Research] → [Human Insight] → [Framework-Guided Analysis] → [Report]
    (Complex-AI)          (Complex-Human)      (Complicated-AI)        (Complicated-AI)
         ↓                      ↓                      ↓
   采集左翼分析           构建理论框架           应用框架深入分析
   形成初步感觉           (洞察与判断)           (结构化分析)
                              ↓
                    【域降解点:Complex → Complicated】
```

**关键特征:域转换模式 (Domain Transformation Pattern)**
- **Complex → Human Intervention → Complicated**
- 人类洞察将Complex问题转化为Complicated问题
- 这是一个关键的降解机制

**MAS设计**:

**阶段1: Conjunctural Overview (Complex-AI)**
- 采集标准化信息:
  - 基本国情
  - 左翼组织材料
  - 学术分析
  - 新闻报道
- 输出:综合材料集(Markdown reports + JSON summaries)
- AI角色:探索和整理,不做判断

**人类介入点 (Complex-Human) - 域降解点**
- 阅读材料
- 与专家交流
- 形成洞察
- **决定理论框架**(如"新殖民主义"视角)
- **输出**:明确表述的理论框架文档

**阶段2: Framework-Guided Analysis (Complicated-AI)**
- 输入:人类确定的理论框架
- 重新采集针对性数据(基于框架的关键概念)
- 应用框架深入分析
- 生成结构化报告
- AI角色:基于明确框架的专家分析

**人机协作模式**:
- **解耦点**: Overview之后,深入分析之前
- **交接内容**: 理论框架的明确表述(文档化)
- **转化效果**: Complex任务→Complicated任务
- **时间解耦**: AI不等待人类,人类不被催促

**具体例子**: 南非阶级分析
- **Complex阶段**:
  - 不能用通用阶级分析框架
  - AI采集左翼组织分析、学术文献、历史资料
- **人类洞察**:
  - 读材料,发现种族问题是核心矛盾
  - 与专家交流,确定"新殖民主义"理论框架
- **Complicated阶段**:
  - 基于"新殖民主义"框架,AI深入分析:
    - 土地所有权结构
    - 经济依附关系
    - 种族与阶级的交织
    - 政治权力分配
- **结果**: 深刻的阶级分析,抓住了南非特殊性

**关键设计原则**:
1. **识别洞察需求点**: 明确哪里需要人类"灵光一闪"
2. **完全解耦**: 不要让AI等待人类,也不要让人类被AI催促
3. **接口清晰化**: 人类输出(理论框架)必须可以明确表述和文档化
4. **域转换利用**: 善用人类洞察将Complex降解为Complicated
5. **降解不可逆**: 一旦框架确定,后续分析进入Complicated,不再回到Complex

**变体**:
- 可能需要多轮迭代:初步框架→试分析→调整框架→深入分析
- 不同研究对象可能需要不同框架(无法批量化)
- 框架本身可能成为理论贡献

**域降解关系**:
- **入口**: 通常从Chaotic或Complex开始
- **降解**: 通过人类洞察降解到Complicated
- **稳定**: 如果框架成熟,可能进一步标准化(但不常见)

#### Exemplar 3b: 编辑判断型任务

**情境**: 需要基于价值判断、美学标准或编辑视角进行选择和组织

**问题**:
- 选择标准无法完全形式化
- 需要识别隐含的关联和模式
- 涉及平衡、连贯、重要性判断
- AI容易遗漏上下文和"讲故事"的需求

**业务架构**:
```
Decoupled Two-Stage Pipeline:
[AI: Candidate Generation] → [Human: Curation] → [AI: Production]
   (Complicated-AI,可选)     (Complex-Human)    (Complicated-AI)
         ↓                        ↓                  ↓
  生成候选项列表              编辑判断选择         基于选择深度制作
  (可带推荐依据)              (模式识别)          (结构化工作)
                                  ↓
                    【域边界:Complex部分不降解,而是解耦】
```

**特殊性:不追求降解,而是隔离**
- 与Exemplar 3a不同,这里的Complex判断**不转化为Complicated**
- 而是**承认AI局限,完全解耦**
- 人类保留Complex部分,AI处理前后的Complicated部分

**MAS设计**:

**阶段1: 候选生成 (Complicated-AI,可选)**
- 从新闻源采集本周所有新闻
- 简单分类和排序
- 提供推荐列表(但不做最终决定)
- 注意:这一步是辅助性的,可有可无

**人类介入点 (Complex-Human) - 不可降解的核心**
- 决定选择哪10-12条新闻
- 识别:
  - 多条新闻实为同一事件
  - 需要合并讲述的关联新闻
  - 重要性权衡
  - 类型和正负面平衡
- **输出**:新闻线索列表(leads) - 简单的标题+URL列表

**阶段2: 深度制作 (Complicated-AI)**
- 输入:人类确定的新闻线索
- **完全独立**,不关心阶段1如何发生
- 对每条线索:
  - 7个角度全方位调研(事件全貌、媒体报道、专家评论、历史背景、专业知识、地缘政治、影响分析)
  - 撰写80字精炼摘要
- 编撰成邮件期刊

**人机协作模式**:
- **完全解耦**: 两阶段完全独立
- **接口最小化**: 新闻线索列表(可以是简单的Markdown列表)
- **人类决策权**: 编辑判断完全由人完成,AI不参与
- **各司其职**: AI做深度调研(Complicated),人类做策展(Complex)

**具体例子**: News on China周刊制作

**为什么编辑判断是Complex且不可降解?**
- **识别"同一事件"**: 四中全会从人事/经济/科技等角度报道,AI难以识别这是同一主题
- **讲故事**: 想把相关但不完全相关的新闻合并,讲一个连贯故事
- **重要性**: 哪种新闻更重要、为什么重要 - 涉及价值判断
- **平衡**: 一期期刊的类型平衡、正负面比例 - 编辑美学

**关键设计原则**:
1. **承认AI局限**: 有些判断AI就是做不好,不要勉强
2. **解耦而非半自动**: 不要试图让AI"辅助"人类判断,而是完全分离
3. **界面最小化**: 人类输出应该尽可能简单(一个列表即可)
4. **发挥各自优势**: AI做深度调研(信息处理能力),人类做模式识别(洞察力)
5. **不追求降解**: 接受某些Complex判断无法自动化

**变体**: 其他策展型任务
- 文献综述中选择关键文献
- 建立研究对象的分类法/本体论
- 确定案例研究的选择标准
- 展览/课程的内容策展

**域降解关系**:
- **不降解**: 这类Complex判断长期保持在Complex
- **隔离**: 通过解耦,让AI处理周边的Complicated任务
- **人机共生**: 形成稳定的人机协作模式,而非追求全自动化

### 3.4 Chaotic域范例

#### Exemplar 4: 创造性田野研究

**情境**: 研究对象缺乏既有文献和数据,需要原创性数据生成

**问题**:
- 现有材料不存在或严重不足
- 因果关系不清晰
- 必须通过实地介入创造新数据
- 理论问题本身尚未明确

**业务架构**:
```
Human-Led Chaotic Process (域降解序列):
[Fieldwork (Human)] → [Data Organization (AI-Clear)] → [Iterative Sense-Making (Human)]
        ↓                        ↓                              ↓
   访谈、观察             转录、整理                    识别模式、提炼理论
   创造新数据             辅助性任务                    (逐步转向Complex)
      ↓                                                        ↓
【Chaotic: Act-Sense-Respond】                    【进入Complex: Probe-Sense-Respond】
                                                               ↓
                                          (后续可能进入Complicated: 应用框架分析)
```

**特征:域降解的起点**
- **开始**: Chaotic(无信息,因果关系不清,必须Act)
- **行动**: 田野调研创造数据(Act)
- **感知**: 在数据积累过程中识别模式(Sense)
- **转变**: 建立初步秩序,进入Complex域(Respond)
- **后续**: 可能继续降解到Complicated域

**MAS设计**:

**主导:人类田野工作 (Chaotic-Human)**
- 实地访谈
- 参与观察
- 文档收集
- 关系建立
- **创造之前不存在的数据**
- Act-Sense-Respond:在行动中调整方向

**辅助:AI处理Clear/Complicated子任务**
- 转录访谈录音 (Clear)
- 整理田野笔记为结构化文档 (Clear/Complicated)
- 初步主题编码 (Complicated)
- 补充文献综述 (Complicated)
- 时间线和事件整理 (Complicated)

**AI不应做**:
- 不能代替数据采集(信息不存在于互联网)
- 不能代替理论构建(模式尚未浮现)
- 不能识别因果关系(因为根本不存在明确的)
- 不能决定调研方向(需要现场判断)

**人机协作模式**:
- **人类绝对主导**: 95%+的核心工作由人完成
- **AI最小化辅助**: 处理机械性子任务,释放人类时间
- **迭代式**: 随着数据积累,逐步明确研究问题和方向
- **灵活性**: 不预设固定流程,根据田野发现调整

**具体例子**: 榕江村超研究

**Chaotic状态的特征**:
- **问题**:数字技术如何助力乡村振兴?
  - 因果机制不清晰
  - GDP无明显变化
  - 旅游收入提高但分配机制未知
  - 流量转化路径不明
- **现状**:现有材料太粗浅,村超是新事物,无深入研究
- **必须Act**:实地调研是唯一获取关键信息的途径

**需要实地探索的关键问题**:
- 月流量实际有多少?
- 流量从哪些媒体/自媒体来?
- 县里如何运作自媒体矩阵?
- 流量如何转化为经济?
- 转化形式和效果?
- 各村寨如何参与创造和转化流量?

**AI辅助的具体方式**:
- 访谈前:整理背景资料、准备访谈提纲
- 访谈后:转录录音(Clear任务)
- 整理阶段:将田野笔记整理为结构化文档
- 分析阶段:初步主题编码,识别关键词频次

**关键设计原则**:
1. **接受人类中心性**: 不要试图让AI主导
2. **识别辅助任务**: 找出可以自动化的Clear部分
3. **保持灵活性**: 不要预设固定流程,田野工作是涌现的
4. **记录涌现过程**: 数据创造过程本身是研究成果
5. **规划域降解**: 思考如何从Chaotic→Complex→Complicated

**Chaotic → Complex的转变标志**:
- ✓ 积累了足够一手数据
- ✓ 开始看到初步模式和关联
- ✓ 可以提出理论性问题
- ✓ 进入Probe-Sense-Respond模式(可以开始"试探"而非只是"行动")

**域降解的后续路径**:
```
Chaotic (田野调研)
  → Complex (识别理论框架,如"流量经济"模式)
  → Complicated (基于框架的结构化分析,如量化各村寨的参与度)
  → 可能产出:理论模型 + 政策建议
```

**变体**:
- **档案研究**: 发掘未被研究的历史档案(创造新数据)
- **口述历史**: 记录即将消失的个人经历(创造新数据)
- **行动研究**: 研究者参与实践并观察(边行动边研究)

**域降解关系**:
- **起点**: Chaotic是研究的起点(当现有知识严重不足时)
- **降解目标**: 通过创造数据,建立初步秩序,进入Complex
- **后续**: 从Complex继续降解到Complicated
- **成果**: 不仅是研究结论,更是知识基础本身的建立

## 附录:四域快速参考表

| 域 | 信息状态 | 因果关系 | AI角色 | 人类角色 | 决策模式 | 典型任务 | Exemplar |
|----|---------|---------|--------|---------|---------|---------|----------|
| **Clear** | 结构化存在 | 明确 | 全自动执行 | 定义任务 | Sense-Categorize-Respond | 下载、转录 | Ex.1 |
| **Complicated** | 存在需解读 | 明确但需分析 | 专家系统 | 定义框架 | Sense-Analyze-Respond | 指标评估 | Ex.2 |
| **Complex** | 存在需整合 | 事后可见 | 辅助材料准备 | 识别模式 | Probe-Sense-Respond | 理论构建、编辑判断 | Ex.3a/3b |
| **Chaotic** | 基本不存在 | 不可辨识 | 最小化辅助 | 创造数据 | Act-Sense-Respond | 田野调研 | Ex.4 |

**域降解的典型路径**:
- Chaotic → Complex: 通过创造数据建立初步秩序
- Complex → Complicated: 通过人类洞察确立理论框架
- Complicated → Clear: 通过标准化固化最佳实践(较少发生)

**关键设计原则**:
1. 识别任务所在的域
2. 规划域降解路径
3. 设计人类介入点
4. 在每个域应用合适的方法
5. 利用AI的长处,尊重人类的不可替代性
